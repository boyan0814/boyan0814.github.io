May 17: Code Review

  • main.py
    • init
      • 初始化各種屬性，包括數據存儲的路徑，並設置其他類別的實例（**TextSplitter、Loader、LLM、VectorDB）。TextSplitter** 用於將大型文檔分成更小的片段進行處理。**Loader** 可能是一個負責加載不同類型文檔（PDF、URL）的類別。**LLM（語言模型）似乎是核心處理組件，使用了一個 OpenAI API 金鑰進行初始化。VectorDB** 用於存儲和檢索表示文本文檔的向量。
    • save_processed_urls : 將先前處理過的 URL 集保存到 JSON 文件中。
    • process_document :
      • 通過使用提供的加載器函數加載文檔進行處理。
      • 如果文檔不為 None 且之前未處理過（根據 self.previous_urls 檢查），則將文檔分成片段，將它們添加到向量數據庫（**self.vdb**），標記 URL 為已處理並保存已更新的已處理 URL 列表。
      • 返回處理後的文檔。
    • pdf_to_doc 和 url_to_doc : 使用 process_document 分別處理 PDF 文件或 URL。
    • qna : 通過查詢向量數據庫（**self.vdb）以檢索相關文本，然後使用語言模型（self.llm**）生成答案，處理一個問題提示。
    • detections : 通過將 URL 轉換為文檔，然後使用語言模型來檢測文檔中的某些內容。
    • custom : 與 detections 類似，但允許指定自定義提示名稱，用於更專業化的查詢。
    • summarize :
      • 通過處理 URL 來生成其內容的摘要，然後使用語言模型。然後將摘要存儲在向量數據庫（**self.vdb**）中，並提取文檔文本中的指示器（威脅指示器）。返回摘要、思維導圖和指示器。
  • llm.py : 使用 OpenAI 的 API 來執行各種任務，例如生成摘要、回答問題等。
  • loader.py : 負責從不同來源加載文檔並轉換為統一格式的文檔。
  • schema.py : 定義了一些 Pydantic 模型，用於定義結構化的數據。
  • iocs.py : 定義了一個 extract_iocs 函式，用於從文本內容中提取指標
  • chunker.py : 定義了一個 TextSplitter 類別，用於將文本分成更小的片段。
  • vectordb.py : 定義了一個 VectorDB 的類別，用於與嵌入式向量資料庫進行交互。

May 31: Final report for what I have learned

  1. 透過 OPENAI 的 API 跑模型

    降低本機運行的負擔，以較低的初始成本啟用服務。

  2. 文字探勘

    將 URL 內文擷取後與指定的 prompt 一同傳給模型擷取出需要的內容，

    實作方式比最近研讀的威脅情資收集方便與快速很多。

    但多來源的威脅情資內容不一致性的問題不確定在這裡要如何處裡。

  3. Vector database

    將探勘後的結果存入向量資料庫，後續聊天時以查閱資料庫內容為主。

    避免使用通用 gpt 查詢出現幻視狀況嚴重的問題。

  • 缺點 :

    前面的範例都是以官方文件的 URL 下去測，也並非任何網頁都適配。

    比如中文內容的部分解析會有些奇怪 :

    【臺灣資安大會直擊】荷蘭網路威脅態勢與防護經驗大公開，荷蘭資安三角洲專家強調資安韌性需擴及各產業供應鏈

    且他只能吃 HTML 上 TEXT 的部分，

    就不行了。比如下列網址 :

    Ivanti Patches Critical Remote Code Execution Flaws in Endpoint Manager

    並且他沒有做編輯 DB 內容的功能，可能要直接進 ChromaDB 操作。

  • 心得 :

    總體來說還是使我感受到 gpt 的強大，在多年前沒有套模型的情況做起來必定費時費力。

    但如今只需要做好爬蟲、前處理、叫 API，就能做好知識倉儲且易於使用。
